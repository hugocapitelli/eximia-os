## PASSO A PASSO PRÁTICO (RECEITA COMPLETA)

### PASSO 1: Defina o Destino (antes de pensar no conteúdo)

#### 1.1 Competência-Alvo
**Formato:** `[VERBO] + [OBJETO] + [CONTEXTO]`

**Exemplos:**
- ❌ "Entender priorização" (vago, passivo)
- ✅ "Negociar prioridades com stakeholder em reunião de 15 min, mantendo alinhamento e prazos viáveis"
- ✅ "Conduzir discovery interview estruturada em 30 min, coletando 80% dos insights-chave"
- ✅ "Analisar viabilidade financeira de projeto em 1h, usando DCF e recomendando go/no-go"

#### 1.2 Evidência (como você sabe que ele aprendeu?)
- Um artefato (documento, plano, protótipo)
- Uma decisão (com justificativa)
- Uma apresentação (pitch, demo)
- Um desempenho (role-play gravado, simulação)

#### 1.3 Critérios de Qualidade (rubrica simples)
Exemplo para "Discovery Interview":
- ✅ Clareza das perguntas (abertas, não indutivas)
- ✅ Profundidade da escuta (follow-ups)
- ✅ Captura de evidências (notas estruturadas)
- ✅ Síntese dos insights (padrões identificados)

#### 1.4 Contexto Real
**Pergunta:** Em que cenário do mundo real essa competência aparece?

Exemplo: "PM júnior recebendo demanda do CEO às 17h de sexta-feira, sem contexto completo, precisando decidir se entra no sprint."

### PASSO 2: Escolha o "Problema-Motor" (a experiência que puxa tudo)

#### O que é um Problema-Motor?
É a **situação inicial** que:
- Tem tensão (tempo curto, recursos limitados, conflito de interesses)
- Exige decisão (o aluno precisa escolher algo)
- Permite feedback observável
- Está no nível certo de dificuldade (desafiador mas possível)

#### Tipos de Problema-Motor

**1. Caso Real (Adaptado)**
- Baseado em situação verídica
- Nomes/dados alterados
- Contexto rico com ruído (informação relevante + irrelevante)

**2. Simulação (Role-Play)**
- Papéis definidos (você é o PM, seu colega é o stakeholder)
- Restrições explícitas (15 min, orçamento X, prazo Y)
- Gravável/observável

**3. Dilema com Trade-offs**
- Sem resposta óbvia
- Todas as opções têm prós e contras
- Força priorização

**4. Desafio Prático**
- Construir/criar algo
- Com restrições (tempo, recursos, critérios)
- Entregável tangível

#### Template de Problema-Motor

```markdown
# Problema-Motor: [Nome do Caso]

## Contexto
[2-3 parágrafos com situação rica]

## Seu Papel
Você é [cargo/função] em [empresa/situação]

## Situação Atual
[O que está acontecendo agora]
[Pressões/urgências]

## Sua Missão
[O que você precisa decidir/fazer]

## Restrições
- Tempo: [X minutos/horas]
- Recursos: [limitações]
- Critérios: [o que conta como sucesso]

## Entregável
Você deve produzir: [artefato específico]
Formato: [documento/apresentação/plano/decisão]
```

### PASSO 3: Desenhe a Aula em Ciclos (NÃO em capítulos)

#### Estrutura de Um Ciclo Completo (1 módulo/aula)

```
┌────────────────────────────────────────────────────────────┐
│                     CICLO EXPERIENCIAL                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. SENTIR (Experiência Concreta) — 18% do tempo          │
│     ├── Apresenta o cenário (problema-motor)               │
│     ├── Pergunta: "O que você faria?"                      │
│     ├── Aluno decide/age sem teoria prévia                 │
│     └── Entregável: Decisão + justificativa breve          │
│                                                            │
│  2. OBSERVAR (Reflexão) — 12% do tempo                     │
│     ├── Pausa para consciência                             │
│     ├── Perguntas: O que priorizou? Por quê?               │
│     ├── Onde teve insegurança?                             │
│     ├── Que suposição fez?                                 │
│     └── Entregável: 3 aprendizados + 1 dúvida real         │
│                                                            │
│  3. PENSAR (Conceitos) — 18% do tempo                      │
│     ├── AGORA entra teoria (como ferramenta)               │
│     ├── Nomeia o que aconteceu                             │
│     ├── Mostra 1-3 modelos (máximo)                        │
│     ├── Compara com o que eles fizeram                     │
│     └── Entregável: "Como conceito muda minha solução?"    │
│                                                            │
│  4. FAZER (Experimentação) — 18% do tempo                  │
│     ├── Aluno refaz ou enfrenta variação                   │
│     ├── Versão 2 (com teoria aplicada)                     │
│     ├── Novo caso com outra restrição                      │
│     └── Entregável: Versão melhorada + execução            │
│                                                            │
│  5. VALIDAR (Feedback) — 12% do tempo                      │
│     ├── Feedback rápido, objetivo, utilizável              │
│     ├── Checklist/rubrica                                  │
│     ├── Peer review guiado                                 │
│     ├── Comparação com boas práticas                       │
│     └── Entregável: "Vou ajustar X, Y e Z"                 │
│                                                            │
│  6. INTERNALIZAR (Transferência) — 22% do tempo            │
│     ├── Transforma em rotina e identidade                  │
│     ├── "Onde vou aplicar isso esta semana?"               │
│     ├── "Que gatilho vai me lembrar?"                      │
│     ├── Diário de aprendizagem (2-5 min)                   │
│     └── Entregável: Compromisso + plano de aplicação       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### PASSO 4: Construa a Trilha do Curso (Macrodesign)

#### Progressão de Módulos

Para um curso completo, você **repete ciclos** e **aumenta complexidade**.

```
MÓDULO 1: Fundamentos + casos simples
├── Problema-motor: cenário básico, 1 variável
├── Decisões binárias (sim/não, A ou B)
└── Feedback direto

MÓDULO 2: Variações + ruído (mais informação, mais pressão)
├── Problema-motor: cenário com ambiguidade
├── Múltiplas variáveis
└── Informação incompleta/conflitante

MÓDULO 3: Casos com conflito humano
├── Problema-motor: stakeholders com agendas ocultas
├── Emoções, política, negociação
└── Trade-offs sociais/éticos

MÓDULO 4: Projeto aplicado no mundo real
├── Problema-motor: situação do trabalho do aluno
├── Autonomia total
└── Consequências reais

MÓDULO 5: Demonstração final + reflexão + plano futuro
├── Apresentação pública (para pares/superiores)
├── Meta-reflexão (o que mudou em mim?)
└── Roadmap de 90 dias
```

#### Progressão em 4 Dimensões

À medida que o curso avança, aumente:

1. **Autonomia do aluno** (de guiado → autodirigido)
2. **Realismo do cenário** (de simulado → real)
3. **Exigência de qualidade** (de "tentativa" → "pronto para produção")
4. **Pressão de tempo/complexidade** (de "sem pressa" → "urgente e ambíguo")

### PASSO 5: Planeje a Logística da Experiência

#### Checklist de Execução

Para cada atividade, defina:

| Item | Especificação |
|------|---------------|
| **Tempo** | Ex.: 10 min / 15 min / 20 min |
| **Formato** | Individual / Dupla / Grupo / Plenária |
| **Materiais** | Brief do caso / Template / Rubrica / Slides |
| **Papéis** | Facilitador / Aluno / Observador / Stakeholder (role-play) |
| **Regras** | Como decidir / Como registrar / Como discutir |
| **Entregável** | O que o aluno produz (sempre 1 página máximo) |

#### Atalho de Ouro
✅ **Sempre tenha um template do entregável (1 página).**

Exemplo para "Discovery Interview":
```
Template: Ficha de Interview
├── Perguntas (5-7 abertas)
├── Notas (observações)
├── Insights (3-5 padrões)
└── Próximos passos (ações)
```

Isso acelera MUITO a execução.

### PASSO 6: Crie Instrumentos de Avaliação Coerentes

#### Rubrica Simples (0-2 ou 0-3 por critério)

**Exemplo: Discovery Interview**

| Critério | 0 - Iniciante | 1 - Intermediário | 2 - Avançado |
|----------|---------------|-------------------|--------------|
| **Clareza da Decisão** | Vaga ou ausente | Clara mas incompleta | Clara, completa, justificada |
| **Coerência com Contexto** | Ignora restrições | Considera algumas | Otimiza para contexto |
| **Justificativa Baseada em Evidência** | Opinião | Algumas evidências | Dados + análise |
| **Viabilidade** | Irrealista | Possível mas difícil | Executável |
| **Ética/Impacto** | Não considerado | Mencionado | Analisado a fundo |
| **Aprendizagem Refletida** | Não reflete | Reflexão superficial | Meta-cognição forte |

**Total:** 0-12 pontos

#### Tipos de Evidência

- **Documento** (plano, relatório, análise)
- **Apresentação** (pitch, demo, walkthrough)
- **Simulação** (role-play gravado)
- **Execução real** (aplicou no trabalho, trouxe resultado)

#### Feedback Formativo vs Somativo

**Formativo (durante):**
- Objetivo: melhorar enquanto aprende
- Formato: checklist rápido, 2 forças + 1 ajuste
- Frequência: a cada etapa do ciclo

**Somativo (final):**
- Objetivo: certificar mastery
- Formato: rubrica completa + projeto final
- Frequência: fim do módulo/curso

#### Estrutura do Roteiro

```markdown

**Competência:** [Verbo + Objeto + Contexto]

## ETAPA 1: SENTIR (18% = [X] min)

### Abertura (2 min)
**Fala do facilitador:**
"Hoje vocês vão decidir se aceitam um projeto com prazo impossível.
Não há resposta certa. Há trade-offs.
Você tem 10 minutos para decidir."

### Apresentação do Caso (5 min)
**Material:** Brief do caso (1 página)
**Instrução:**
"Leia o caso. Identifique:
1. O dilema principal
2. As restrições
3. O que está em jogo"

### Tomada de Decisão (10 min)
**Formato:** Individual
**Entregável:** Template de Decisão (1 página)
├── Decisão: [Aceitar/Recusar/Negociar]
├── Justificativa: [3 razões]
└── Riscos assumidos: [2-3]

**Pergunta de fechamento:**
"Onde você sentiu mais tensão ao decidir?"

---

## ETAPA 2: OBSERVAR (12% = [X] min)

### Debrief Guiado (10 min)
**Perguntas (uma por vez, deixa silêncio de 30 seg):**

1. "O que você priorizou na decisão? Por quê?"
2. "O que você deixou de fora? Por quê?"
3. "Onde você teve insegurança?"
4. "Que suposição você fez sem validar?"

**Formato:** Plenária ou grupos pequenos

### Captura de Insights (5 min)
**Entregável:** Post-its ou Google Doc colaborativo
├── 3 aprendizados
└── 1 dúvida real

---

## ETAPA 3: PENSAR (18% = [X] min)

### Mini-Aula Conceitual (15 min)
**Slides:** Máximo 5 (1 ideia por slide)

**Conceitos-chave:**
1. [Framework X]: quando usar, como aplicar
2. [Trade-off Y]: custo vs benefício
3. [Antipadrão Z]: erro comum

**Comparação:**
"Vamos ver como o framework teria mudado sua decisão..."

**Exemplos curtos:**
├── Caso A (sucesso)
├── Caso B (fracasso)
└── Contraexemplo

### Conexão com Prática (3 min)
**Pergunta:**
"Como esse conceito muda sua solução inicial?"

**Entregável:** 1 parágrafo no template

---

## ETAPA 4: FAZER (18% = [X] min)

### Refazer com Teoria (15 min)
**Instrução:**
"Agora refaça sua decisão aplicando [Framework X].
Compare:
├── Versão 1 (sem framework)
└── Versão 2 (com framework)"

**Formato:** Individual ou dupla

**Entregável:** Versão 2 melhorada no template

### Execução Prática (opcional, 10 min)
**Simulação:** Role-play da reunião de negociação
├── Pessoa A: PM (você)
├── Pessoa B: Stakeholder (script fornecido)
└── Observador: Anota decisões

---

## ETAPA 5: VALIDAR (12% = [X] min)

### Peer Review (8 min)
**Instrução:**
"Troque seu template com um colega.
Use a rubrica para dar:
├── 2 forças (o que está forte)
└── 1 ajuste (o que melhoraria)"

**Material:** Rubrica 0-2 (fornecida)

### Autoavaliação (4 min)
**Pergunta:**
"Baseado no feedback, o que você vai ajustar?"

**Entregável:** Lista de 3 ajustes

---

## ETAPA 6: INTERNALIZAR (22% = [X] min)

### Plano de Aplicação (10 min)
**Template de Compromisso:**
├── Onde vou aplicar isso esta semana? [situação específica]
├── Que gatilho vai me lembrar? [evento/horário]
├── Qual hábito mínimo posso sustentar? [ação de 5 min]
└── Como vou medir sucesso? [evidência]

### Diário de Aprendizagem (5 min)
**3 Prompts:**
1. O que aprendi hoje que não sabia antes?
2. O que ainda me incomoda ou não entendi?
3. Como isso muda o que eu fazia antes?

**Formato:** Individual, escrito

### Fechamento Ritual (3 min)
**Fala do facilitador:**
"Vocês decidiram sob pressão, ajustaram com feedback, e agora têm um plano.
Compromisso de 7 dias: apliquem e voltem com evidência."

**Próximo encontro:**
Check-in de 5 min: "Quem aplicou? O que aconteceu?"
```

## PRINCÍPIOS DE DESIGN (REGRAS DE OURO)

### 1. Não pule "SENTIR"
❌ Começar com teoria
✅ Começar com tensão/dilema

**Por quê:** É o "Sentir" que dá energia e sentido para o resto.

### 2. Teoria = Ferramenta, não Fim
❌ "Aqui está o framework, agora aplique"
✅ "Você decidiu X. Veja como o framework teria ajudado."

### 3. Sempre Tenha Entregável por Etapa
Cada etapa deve produzir algo observável (1 página, 1 decisão, 1 artefato).

### 4. Feedback Rápido e Utilizável
❌ "Bom trabalho!"
✅ "Clareza da decisão: forte. Viabilidade: ajustar prazos (muito apertados)."

### 5. Compromisso > Intenção
❌ "Vou tentar aplicar"
✅ "Segunda às 10h, na reunião com CEO, vou usar checklist de priorização"

## TENSÃO COMO MOTOR DA AULA

### Tipos de Tensão

**1. Dilema Moral/Ético**
"Ganhar o contrato com propina ou perder para concorrente?"

**2. Trade-off de Recursos**
"Qualidade, prazo ou escopo — escolha 2 de 3"

**3. Conflito de Interesses**
"CEO quer feature X, usuário precisa de Y, engenharia diz que Z é viável"

**4. Risco vs Reward**
"Arriscar tudo em pivot ou iterar com segurança?"

**5. Urgência + Ambiguidade**
"Decidir em 10 min com 30% da informação"

### Fórmula da Tensão

```
Tensão = Pressão (tempo/recursos) × Ambiguidade (incerteza) × Stakes (o que está em jogo)
```

**Exemplo de alta tensão:**
"Você tem 15 min para decidir se cancela o evento de 500 pessoas (investimento R$ 200k) porque o palestrante principal desistiu. CEO está esperando sua resposta."

## HEUTAGOGIA: AUTONOMIA REAL DO APRENDIZ

### 4 Escolhas Reais para Incluir

1. **Escolher qual caso resolver**
   Ex.: "Caso A (iniciante), Caso B (intermediário), Caso C (avançado)"

2. **Escolher critérios de sucesso**
   Ex.: "Você define o que conta como 'boa solução' (3-5 critérios)"

3. **Escolher formato de entrega**
   Ex.: "Apresente como: texto / áudio / mapa / checklist / vídeo"

4. **Escolher nível do desafio**
   Ex.: "Básico (guiado), Avançado (autônomo)"

### Princípio
**O aprendiz que escolhe, se responsabiliza.**

## BLOCOS ATIVÁVEIS (não Tópicos)

### Estrutura de Bloco

```
BLOCO = [HABILIDADE/DECISÃO] + [PERGUNTA DE ATIVAÇÃO]
```

**Exemplos:**

❌ Tópico: "Capítulo 3: Priorização"
✅ Bloco Ativável: "Priorização sob pressão" + "O que você corta quando tudo é urgente?"

❌ Tópico: "Capítulo 5: Comunicação"
✅ Bloco Ativável: "Negociação de escopo" + "Como você diz 'não' sem perder o cliente?"

❌ Tópico: "Capítulo 7: Análise de dados"
✅ Bloco Ativável: "Decisão data-driven" + "Que métrica prova que vale a pena?"

### Checklist de Bloco Ativável

- [ ] Tem uma pergunta forte de abertura?
- [ ] Exige decisão ou ação?
- [ ] Pode ser praticado em 20-30 min?
- [ ] Tem feedback observável?

## AVALIAÇÃO COM EVIDÊNCIA (Kirkpatrick 4 Níveis)

### Nível 1: Reação (percepção de valor)
**Pergunta:** "Você achou relevante?"
**Medição:** Survey pós-aula (NPS, satisfação)

### Nível 2: Aprendizagem (o que consegue demonstrar)
**Pergunta:** "Você consegue fazer?"
**Medição:** Rubrica de desempenho, teste prático

### Nível 3: Comportamento (aplicação no trabalho/vida)
**Pergunta:** "Você está usando?"
**Medição:** Follow-up 30-90 dias, observação, relatório de aplicação

### Nível 4: Impacto (resultados de negócio/vida)
**Pergunta:** "Está gerando resultado?"
**Medição:** KPIs de negócio, ROI, mudança mensurável

### Template de Avaliação 360

```markdown
# Avaliação do Módulo [X]

## Nível 1: Reação (imediato)
- [ ] Survey de satisfação (5 perguntas)
- [ ] NPS: "Recomendaria para colega?"

## Nível 2: Aprendizagem (fim da aula)
- [ ] Rubrica de desempenho (0-2 por critério)
- [ ] Entregável final (artefato produzido)

## Nível 3: Comportamento (30 dias depois)
- [ ] Check-in: "Aplicou no trabalho?"
- [ ] Evidência: Print/screenshot/relato

## Nível 4: Impacto (90 dias depois)
- [ ] Mudança em KPI relevante
- [ ] Relato de resultado (caso de sucesso)
```

## PROTOTIPAÇÃO ITERATIVA

### Antes de Lançar: Rode Uma Versão Curta

**Checklist de Prototipação:**

1. **Teste o dilema**
   - [ ] Alguém consegue entender em 2 min?
   - [ ] Gera tensão real (não é óbvio)?
   - [ ] Tem informação suficiente para decidir?

2. **Veja se a reflexão gera insights**
   - [ ] As perguntas do debrief são poderosas?
   - [ ] Os alunos trazem aprendizados não-óbvios?

3. **Corte teoria se estiver longa**
   - [ ] Teoria cabe em 15 min?
   - [ ] É mínimo necessário para agir?

4. **Fortaleça o "Fazer" (transferência)**
   - [ ] A versão 2 é significativamente melhor que versão 1?
   - [ ] O plano de aplicação é específico?

### Ciclo de Melhoria

```
V1 (protótipo)
  ↓ (testar com 3-5 pessoas)
Feedback
  ↓ (ajustar)
V2 (melhorada)
  ↓ (testar com turma piloto)
Feedback
  ↓ (refinar)
V3 (pronta para escala)
```

## CONCLUSÃO

### O Segredo do Método (em 1 frase)

> **"Conteúdo entra como ferramenta de decisão e ação — não como ponto de partida."**

### Checklist de Validação Experiencial

Um curso/aula é verdadeiramente experiencial se:

- [ ] Começa com SITUAÇÃO, não com explicação
- [ ] Aluno DECIDE/AGE antes de aprender teoria
- [ ] Teoria entra como FERRAMENTA (não como fim)
- [ ] Cada etapa tem ENTREGÁVEL observável
- [ ] Feedback é UTILIZÁVEL (não só "bom/ruim")
- [ ] Tem TENSÃO real (não é óbvio)
- [ ] Inclui AUTONOMIA (escolhas reais)
- [ ] Termina com APLICAÇÃO no mundo real (não só "entendi")
