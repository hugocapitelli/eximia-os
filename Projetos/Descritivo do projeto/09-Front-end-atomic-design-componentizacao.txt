1. METODOLOGIA ATOMIC DESIGN
================================================================================

O HARVEN.AI atualmente NÃO segue Atomic Design de forma estruturada. Os 
componentes estão organizados de forma ad-hoc na pasta components/, sem 
hierarquia clara de Atoms → Molecules → Organisms → Templates → Pages.

Este documento propõe uma reorganização seguindo Atomic Design (Brad Frost).

================================================================================
5. REGRAS DE COMPONENTIZAÇÃO
================================================================================

5.1. PRINCÍPIOS
--------------

- Um componente por arquivo
- Props tipadas com TypeScript
- Export default quando possível
- Barrels (index.ts) para exports organizados
- Testes junto com componente (quando aplicável)

5.2. NOMENCLATURA
----------------

- PascalCase para componentes
- camelCase para props e variáveis
- Nomes descritivos e específicos
- Evitar abreviações desnecessárias

5.3. ESTRUTURA DE PASTA
----------------------

Cada componente em sua própria pasta:
ComponentName/
├── ComponentName.tsx
├── ComponentName.test.tsx
├── ComponentName.stories.tsx (opcional, Storybook)
└── index.ts

5.4. DEPENDÊNCIAS
-----------------

- Atoms não dependem de outros componentes
- Molecules podem usar Atoms
- Organisms podem usar Molecules e Atoms
- Templates podem usar Organisms, Molecules e Atoms
- Pages podem usar tudo

================================================================================
6. MELHORIAS RECOMENDADAS
================================================================================

6.1. IMEDIATAS
-------------

1. Reorganizar componentes em estrutura Atomic Design
2. Criar barrels (index.ts) para exports
3. Remover componentes deprecated (Navbar)
4. Consolidar componentes similares

6.2. MÉDIO PRAZO
---------------

1. Criar Storybook para documentação de componentes
2. Adicionar testes para todos os componentes
3. Criar design system documentado
4. Implementar variantes consistentes

6.3. LONGO PRAZO
---------------

1. Migrar para biblioteca de componentes (ex: shadcn/ui)
2. Implementar theming avançado
3. Criar componentes acessíveis (WCAG 2.1 AA)
4. Otimizar bundle size (tree-shaking)
